ğŸ¤” What is Docker?

Docker is like a shipping container for your applications. Just like how shipping containers can carry different goods but fit on any ship, Docker containers can hold your applications and run on any computer that has Docker installed.

ğŸš¢ Real-World Analogy

Imagine you're moving to a new house. Instead of packing each item separately, you put everything in standardized boxes. These boxes can be loaded onto any moving truck, regardless of the truck's brand or size. Docker works the same way - it packages your application with everything it needs into a "container" that can run anywhere.

ğŸ’¡ Why Use Docker?

ğŸ”§ Consistency

"It works on my machine" becomes "It works everywhere!" Your app runs the same way on your laptop, your colleague's computer, and production servers.

âš¡ Easy Setup

No more complex installation guides. Just run one command, and your application is ready to go with all its dependencies.

ğŸ”„ Isolation

Each application runs in its own container, so they don't interfere with each other. Like having separate apartments in a building.

ğŸ“¦ Portability

Move your application between development, testing, and production environments without worrying about compatibility issues.

ğŸ“š Basic Docker Terminology

ğŸ—ï¸ Dockerfile

A recipe file that tells Docker how to build your application. Like a cooking recipe with step-by-step instructions.

ğŸ“¸ Image

A snapshot or template of your application. Think of it as a blueprint or a photo that can't be changed.

ğŸ“¦ Container

A running instance of an image. Like the difference between a house blueprint (image) and an actual house you live in (container).

ğŸª Docker Hub

A cloud-based library where you can find and share Docker images. Like an app store for Docker containers.

ğŸš¢ Registry

A place to store and distribute Docker images. Docker Hub is the most popular public registry.

ğŸ—ï¸ Docker Architecture

Docker uses a client-server architecture. The Docker client talks to the Docker daemon, which builds, runs, and manages your containers. It's like having a construction foreman (daemon) who follows blueprints (images) to build houses (containers).

ğŸ—ï¸ Real-World Example: Angular App

Let's break down the Angular application Dockerfile you created. This is a great example of a multi-stage build:

ğŸ“„ The Dockerfile Explained

ğŸ”§ Stage 1: Building the App

FROM node:24 AS builder WORKDIR /app COPY package*.json ./ RUN npm install -g @angular/cli && npm install COPY . . RUN ng build --configuration=production
  • FROM node:24 AS builder - Start with Node.js version 24, call this stage "builder"
  • WORKDIR /app - Set working directory inside container
  • COPY package*.json ./ - Copy package files first (for better caching)
  • RUN npm install... - Install Angular CLI and dependencies
  • COPY . . - Copy all source code
  • RUN ng build... - Build the Angular app for production

ğŸŒ Stage 2: Serving the App

FROM nginx:alpine COPY nginx.conf /etc/nginx/conf.d/default.conf COPY --from=builder /app/dist/angular_first_app/browser /usr/share/nginx/html EXPOSE 80 CMD ["nginx", "-g", "daemon off;"]
  • FROM nginx:alpine - Use lightweight NGINX web server
  • COPY nginx.conf... - Copy custom NGINX configuration
  • COPY --from=builder... - Copy built files from previous stage
  • EXPOSE 80 - Tell Docker this container uses port 80
  • CMD [...] - Start NGINX when container runs

ğŸ’¡ Why Multi-Stage Build?

We use two stages because we need Node.js to build the Angular app, but we only need NGINX to serve it. This keeps the final image small and secure by excluding build tools from the production container.

âš™ï¸ The NGINX Configuration

server { listen 80; server_name localhost; root /usr/share/nginx/html; index index.html; location / { try_files $uri $uri/ /index.html; } }

This configuration ensures that Angular's routing works correctly by redirecting all requests to index.html.

ğŸŒ Sharing on Docker Hub

ğŸ”¨ Building and Running Your Container

1. Build the Image

docker build -t angular-first-app .

This creates an image called "angular-first-app" using the Dockerfile in the current directory (.).

2. Run the Container

docker run -p 8080:80 angular-first-app

This starts a container and maps port 8080 on your computer to port 80 in the container.

ğŸ“¤ Pushing to Docker Hub

1. Login to Docker Hub

sudo docker login

Enter your Docker Hub username and password.

2. Tag Your Image

sudo docker tag angular-first-app YOUR_DOCKERHUB_USERNAME/angular-first-app

This gives your image a name that includes your Docker Hub username.

3. Push to Docker Hub

sudo docker push YOUR_DOCKERHUB_USERNAME/angular-first-app

This uploads your image to Docker Hub where others can access it.

ğŸ‰ Success!

Now anyone in the world can run your Angular application with just one command:
